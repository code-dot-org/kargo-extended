# 0002 Implementation Plan

## Goal

- Deliver the host-side StepPlugin system in `kargo-extended`.
- Keep plugin runtimes out of this repo. This repo owns host runtime,
  discovery, transport, agent execution, and docs.
- Prove the host slice end to end with a minimal `mkdir` plugin.
- Keep OpenTofu and `send-message` easy to add later without implementing
  their real plugin runtimes in this first slice.
- Use `kargo-extended.code.org` for new invented plugin surface.
- Keep existing Akuity API groups where we are matching existing APIs.
- Keep internal promotion types out of plugin RPC. Use dedicated wire types:
  - `StepExecuteRequest`
  - `StepExecuteResponse`
- Public enablement is env var `STEP_PLUGINS_ENABLED` through the chart's
  existing `controller.env`.

## Merge Conflict Discipline

- Follow the root `AGENTS.md` rules.
- In this plan that means:
  - keep real logic under `extended/`
  - keep files outside `extended/` to thin bridge edits
  - keep seam-protection tests under `extended/`
  - after the feature is green, run the post-green upstream-diff pass
- This is a balance, not an absolute. Do not build large parallel subsystems
  under `extended/` just to dodge a small upstream edit.

## Implementation Phases

### Phase 1: Authoring Manifest And Build Tool

- Add Kargo StepPlugin spec types under `extended/pkg/argoworkflows/...`.
- Keep Argo's typed authoring shape for `plugin.yaml`.
- Add `kargo step-plugin build DIR`.
- Build:
  - load `plugin.yaml`
  - optionally embed single `server.*`
  - write `<name>-step-plugin-configmap.yaml`
- Keep the Argo sidecar validation shape.
- Use the first declared container port as the plugin RPC port.

### Phase 2: Discovery And Spec Parsing

- Define Kargo discovery `ConfigMap` parsing and emitting with:
  - Kargo label key/value
  - Argo-shaped sidecar keys
  - Kargo `steps.yaml`
- Generated `ConfigMap` names use `-step-plugin`.
- Parse:
  - `sidecar.automountServiceAccountToken`
  - `sidecar.container`
  - `steps.yaml`
- Reject:
  - invalid sidecar config
  - missing step kinds
  - duplicate step kinds

### Phase 3: Runtime Plugin Registry

- Add Kargo-owned discovery code under `extended/pkg/stepplugin/`.
- Use `STEP_PLUGINS_ENABLED` as the public gate.
- Watch labeled plugin `ConfigMap`s in:
  - the Project namespace
  - `SYSTEM_RESOURCES_NAMESPACE`
- Resolve effective plugins by plugin name before step registration.
- Project namespace wins over `SYSTEM_RESOURCES_NAMESPACE`.
- Build a resolved step registry keyed by step kind.
- Reject collisions with builtin step kinds and other effective plugins.
- Keep this resolver in `extended/`.

### Phase 4: Agent-Backed Promotion Execution

- Keep builtin-only Promotions on the current local engine.
- Any Promotion with a plugin step uses the agent path.
- The controller continues to own:
  - step ordering
  - expression evaluation
  - shared state
  - retries and backoff
  - `Promotion` status writes
- The agent owns:
  - the shared workdir
  - builtin step execution inside the pod
  - plugin step RPC calls to sidecars
- Add agent pod construction with:
  - one main agent container
  - one sidecar per plugin actually used by the Promotion
  - one shared `emptyDir`
  - per-plugin token mount pattern copied from Argo

### Phase 5: Plugin Transport And Agent Runtime

- Add a Kargo-owned plugin RPC layer for `POST /api/v1/step.execute`.
- Define:
  - `StepExecuteRequest`
  - `StepExecuteResponse`
- Use normal JSON on the wire.
- Do not base64-encode step config on the wire.
- Adapt between internal promotion types and plugin RPC wire types at the host
  boundary.
- Copy Argo's bearer-token RPC auth shape with Kargo paths:
  - init container writes one random token per plugin sidecar container name
  - init and main mount `/var/run/kargo`
  - agent main reads `/var/run/kargo/<sidecar-container-name>/token`
  - each plugin sidecar mounts only its own subpath at `/var/run/kargo`
  - plugin sidecar reads `/var/run/kargo/token`
  - agent sends `Authorization: Bearer <token>`
  - bad auth returns `403`
- Keep this separate from optional Kubernetes service account token mounts.

### Phase 6: First Deliverable Boundary

- Deliver in this repo:
  - host runtime
  - discovery/config parsing
  - resolved registry
  - agent path
  - `mkdir` example plugin source files
  - `mkdir` generated `ConfigMap`
  - test fixtures and sample plugin manifests
- Deliver product docs under:
  - `extended/docs-site/05-kargo-external/`
- Expose those docs with one symlink:
  - `docs/docs/05-kargo-external ->
    ../../extended/docs-site/05-kargo-external`
- Keep the generated docs section before `Home`.
- Reach roughly Argo executor plugin doc depth by copying and rewriting where
  that keeps work small.
- Prove the minimal `mkdir` example end to end:
  - `kargo step-plugin build .`
  - install into `kargo-system-resources` or a Project namespace
  - discovery
  - `Stage` step execution
- Do not implement real OpenTofu or `send-message` plugin runtimes in this repo
  in this first host slice.

### Phase 7: Repo Smoke-Test Harness

- Reuse the repo's existing CLI e2e harness:
  - `pkg/cli/tests/e2e.sh`
- Do not fork or duplicate that script for StepPlugins.
- Add fork-owned smoke coverage under:
  - `extended/tests/e2e_stepplugins.sh`
- Keep the outside-`extended/` hook thin:
  - one unconditional source or call from `pkg/cli/tests/e2e.sh`
- Reuse the existing e2e shell helpers, login flow, and cluster assumptions
  instead of rebuilding them in a second script.
- Keep the first smoke target small:
  - one real `mkdir` StepPlugin proof is enough for this slice
- In that smoke flow, prove:
  - `kargo step-plugin build` works on the documented example
  - the generated `ConfigMap` installs and is discovered
  - a `Stage` can run `uses: mkdir`
  - a later builtin step sees the directory the plugin created in shared
    `/workspace`

### Phase 8: Post-Green External Diff Minimization

1. Get the feature green first.
   Start from a working tree where the relevant tests already pass. Do not do
   this phase first.
2. Use the real upstream Kargo history as the comparison base.
   Preferred remote is `upstream`. Preferred branch is `upstream/main`. If
   `upstream` does not exist, add the Akuity Kargo remote first.
3. Fetch before reviewing diffs.
   Run `git fetch upstream` so the comparison base is current enough.
4. Review every file outside `extended/` that was edited for this feature.
   For each one:
   - diff it against `upstream/main`
   - count changed lines
   - count non-contiguous edit blocks
5. Re-evaluate each outside-`extended/` edit now that the full feature works.
   Ask:
   - which edits were only convenient while building, but are no longer needed
   - which edits can move behind helpers or adapters under `extended/`
   - whether a different whole-file strategy collapses two blocks into one or
     removes a block entirely
6. Optimize for fewer merge-conflict surfaces, not prettier upstream-owned
   files.
   Prefer one small ugly bridge over a broad pretty rewrite.
7. Balance diff reduction against compatibility risk.
   Do not create large duplicate subsystems under `extended/` just to avoid a
   small upstream edit. If avoiding an upstream edit requires substantial
   parallel orchestration or behavior that must track upstream closely, the
   upstream edit may be the safer choice.
8. After every cleanup pass on an outside-`extended/` file:
   - rerun the matching tests under `extended/`
   - rerun any broader targeted tests affected by that seam
   - if the cleanup makes the diff worse or the code less obvious, back it out
9. End this phase only when:
   - no obvious helper extraction remains
   - no obvious edit block can be collapsed or removed
   - the matching `extended/` tests are green again

## Tests

Rule:

- if a feature seam forced an edit outside `extended/`, at least one test for
  that seam must live under `extended/`
- those tests are the safety net for future merge-conflict repair in the
  upstream-owned bridge file

Build tool:

- valid `plugin.yaml` builds
- invalid manifest fails validation
- optional `server.*` is embedded
- generated filename uses `-step-plugin-configmap.yaml`
- generated `ConfigMap` name uses `-step-plugin`
- generated label key/value is correct
- generated namespace is preserved from `plugin.yaml`
- first declared container port is preserved for RPC
- `plugin.yaml` -> generated `ConfigMap` -> parsed discovery object round-trip

Discovery and registry:

- valid `ConfigMap`
- invalid sidecar
- invalid `steps.yaml`
- duplicate step kinds
- Project namespace wins by plugin name
- plugin step kind collisions are rejected
- builtin collisions are rejected

Transport:

- `200` returns result
- `403` bad auth fails hard
- `404` caches unsupported method
- `503` retries
- connection refused retries
- timeout retries
- other `4xx/5xx` fail hard
- bearer token is sent on plugin RPC
- `step.config` is normal JSON on the wire, not base64

Enablement and controller behavior:

- when StepPlugins are disabled, the controller ignores plugin discovery
- builtin-only Promotions still use current behavior when StepPlugins are
  disabled
- builtin-only Promotion stays local
- plugin-backed Promotion uses agent path
- timeout lookup uses resolved registry metadata

Agent behavior:

- builtin and plugin steps share one workdir
- plugin sidecars and token mounts are added correctly
- token file paths are `/var/run/kargo/<sidecar-container-name>/token` in
  agent main and `/var/run/kargo/token` in the sidecar
- agent dispatches plugin steps to the right sidecar

## Assumptions

- `StepPlugin` is build input, not a served CRD, in v1.
- StepPlugins are on by default and can be disabled publicly through env var
  `STEP_PLUGINS_ENABLED`, usually set by the chart's existing
  `controller.env`.
- We keep Argo's local-source `server.*` embedding behavior in v1.
- Host v1 in this repo does not also ship the real plugin implementations.
- OpenTofu and `send-message` must stay easy to add later.
- UI schema discovery stays out of the first implementation slice.
- Webhook validation against discovered plugin steps stays out of v1.
- No refresh helper script unless the copy set grows enough to justify it.
